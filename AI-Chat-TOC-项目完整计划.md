# AI Chat TOC — 项目完整计划

## 一、项目定位

一句话描述：**为 AI 聊天页面提供对话目录导航，让用户在长对话中快速找到任意问答。**

目标用户：每天深度使用 ChatGPT / Claude / Gemini 的知识工作者、开发者、学生、研究人员。

核心价值：**长对话场景下的效率工具**——用户不再需要疯狂往上翻滚才能找到之前的问答。


---

## 二、开发阶段

### 阶段一：MVP 验证（第 1-2 周）

目标：完成核心功能，自己用起来，确认痛点真实存在。

**需要做的事：**

1. 实际安装到 Chrome，在 ChatGPT 和 Claude 上测试，修复 DOM 选择器（这些平台的 DOM 结构经常变，需要打开 F12 对照当前页面调整）
2. 验证流式输出时目录更新是否正常（MutationObserver + 防抖逻辑）
3. 验证 SPA 路由切换（切换不同对话时）目录是否正确刷新
4. 测试 Octotree 风格的悬停交互：触发区域是否好触发、收回延迟是否合适、钉住功能是否正常
5. 修复各种边界情况：空对话、超长消息、包含代码块的消息、包含图片的消息
6. Gemini 平台适配和测试

**技术要点：**

- DOM 选择器是这个插件的生命线，每个平台需要维护一份选择器映射，建议写个简单的选择器测试脚本
- 注意 content script 注入时机，有些平台用了 Shadow DOM，需要特殊处理
- 处理好消息文本提取逻辑：用户消息可能包含 Markdown、代码、图片、文件附件等

**完成标志：** 自己日常使用一周，感觉好用，没有明显 bug。


### 阶段二：功能完善（第 3-4 周）

目标：打磨体验，为公开发布做准备。

**需要做的事：**

1. **对话导出** — 将当前对话导出为 Markdown 文件（这是用户呼声很高的功能，也是付费功能的基础）
2. **键盘快捷键** — 除了 Ctrl+Shift+T 切换面板，增加上下箭头在目录项之间跳转、Enter 跳转到对应消息
3. **当前位置追踪** — 滚动页面时，目录中自动高亮当前可见的对话位置（类似文档目录的滚动同步）
4. **记住滚动位置** — 切换对话再切回来时，记住上次看到的位置
5. **国际化** — 至少支持中文和英文界面（目标用户是全球的）
6. **设置页面** — 让用户自定义触发区域宽度、收回延迟时间、侧边栏宽度等

**技术要点：**

- 滚动位置追踪用 IntersectionObserver 实现，比监听 scroll 事件性能好
- 国际化方案用 Chrome 原生的 `chrome.i18n`，在 `_locales` 目录下管理多语言文件
- 设置保存用 `chrome.storage.sync`，这样用户换设备也能同步配置


### 阶段三：高级功能 / 付费功能（第 5-8 周）

目标：构建付费版的功能差异点。

**免费版保留：**
- 对话目录导航（核心功能永远免费）
- 搜索对话内容
- 悬停触发 + 钉住
- 支持 3 个主流平台

**Pro 版功能：**

1. **AI 对话摘要** — 调用 AI API 对每轮对话生成一行摘要，显示在目录中，比只看问题标题信息量大得多
2. **对话导出为多种格式** — Markdown、PDF、Notion 格式、HTML
3. **对话收藏和标签** — 给重要对话打标签，跨会话收藏管理
4. **全文搜索增强** — 支持正则匹配、按时间范围筛选
5. **多平台支持** — Poe、Perplexity、DeepSeek、Kimi 等
6. **自定义主题** — 侧边栏配色、字体大小自定义

## 三、发布计划

### 第一次公开发布（第 3 周末）

**发布到 Chrome Web Store：**

1. 准备 Chrome Web Store 开发者账号（一次性 $5 注册费）
2. 准备商店素材：
   - 128x128 图标（已有）
   - 1280x800 宣传大图（展示侧边栏效果的截图，加上文字说明）
   - 640x400 小宣传图
   - 至少 2 张实际使用截图（ChatGPT 上的效果、Claude 上的效果）
   - 详细的功能描述（中英文）
3. 提交审核（通常 1-3 个工作日）

**同步准备：**

- 创建 GitHub 仓库，开源免费版代码
- 写一个好的 README（含 GIF 动图展示使用效果）
- 搭建一个简单的 Landing Page（用 Next.js 或者纯 HTML，部署到 Vercel）

### 后续发布节奏

- 每 1-2 周发布一个小版本更新
- 每次更新在 changelog 里说明改了什么
- Chrome Web Store 更新审核通常比首次快

## 五、变现模式

### 定价策略

| 方案 | 价格 | 目标 |
|------|------|------|
| Free | $0 | 获客，建立口碑 |
| Pro 月付 | $3.99/月 | 低门槛转化 |
| Pro 年付 | $29.99/年（约 $2.5/月） | 鼓励长期订阅 |
| Lifetime | $59.99（限时） | 早期支持者，快速回收 |

**为什么这个价格：**

- 目标用户已经在为 ChatGPT Plus ($20/月) 或 Claude Pro ($20/月) 付费，$3.99 对他们来说很便宜
- 同类工具（如 ChatGPT 导出插件）的定价区间也在 $3-8/月

### 支付集成

推荐用 **Lemon Squeezy** 或 **Paddle**，原因：

- 它们作为 Merchant of Record 帮你处理全球税务
- 支持信用卡和 PayPal
- 提供 License Key 管理，Chrome 插件可以通过验证 License Key 来激活 Pro 功能
- 集成简单，比 Stripe 自己处理税务省心得多

### 付费验证流程

1. 用户在 Landing Page 购买，获得 License Key
2. 在插件设置页面输入 License Key
3. 插件调用你的验证 API（可以用 Lemon Squeezy 的 API）确认有效性
4. 验证通过后解锁 Pro 功能，状态存到 `chrome.storage.sync`

## 六、技术架构演进

### 现在（MVP）

```
纯前端，无后端
content.js → 直接操作 DOM
配置存 localStorage / chrome.storage
```

### 中期（加付费功能后）

```
前端: content.js + popup + options page
后端: 一个轻量 API (Cloudflare Workers 或 Vercel Edge Functions)
  ├── License Key 验证
  └── 用量统计
支付: Lemon Squeezy
```

### 长期（如果规模增长）

```
前端: 用 Vite + CRXJS 重构，TypeScript 化
后端: 
  ├── License 验证
  ├── AI 摘要服务（代理调用 OpenAI/Anthropic API）
  ├── 云端收藏同步
  └── 用户分析
数据库: PlanetScale 或 Supabase
```

### 核心指标

| 指标 | 追踪方式 | 目标（3个月） |
|------|----------|--------------|
| 安装量 | Chrome Web Store 后台 | 5,000+ |
| DAU（日活） | 插件内匿名统计 | 1,000+ |
| 评分 | Chrome Web Store | 4.5+ 星 |
| Pro 转化率 | Lemon Squeezy 后台 | 3-5% |
| MRR（月收入） | Lemon Squeezy 后台 | $500+ |

### 怎么做匿名统计

在插件内做非常轻量的匿名统计（不收集任何用户数据，只统计事件次数）：

- 插件被激活的次数
- 目录被打开的次数
- 跳转功能使用次数
- 哪个平台使用最多

可以用 Google Analytics 4 的 Measurement Protocol，或者更隐私友好的 Plausible / Umami。

## 八、风险和应对

### 风险 1：平台 DOM 结构频繁变更

这是最大的风险。ChatGPT 和 Claude 都会频繁更新前端代码。

**应对：**

- 建立 DOM 选择器的自动化测试（写一个 Playwright 脚本，定期访问各平台检测选择器是否有效）
- 选择器尽量用 data-testid 等稳定属性，避免依赖 class name
- 在插件内加一个 "反馈" 按钮，用户可以一键报告 "目录不显示了"
- 保持快速响应，DOM 变了尽量 24 小时内修复发版

### 风险 2：平台官方做了类似功能

ChatGPT 或 Claude 随时可能自己加一个对话目录功能。

**应对：**

- 不要只做目录导航，尽快扩展到 "AI 对话管理" 的更大场景（导出、收藏、标签、摘要）
- 做跨平台统一体验，这是官方永远不会做的（ChatGPT 不会帮你管理 Claude 的对话）
- 保持速度优势，小团队迭代快

### 风险 3：Chrome 插件审核政策变化

**应对：**

- 不要过度申请权限，只用必要的 permissions
- 同步开发 Firefox 版本（WebExtension API 大部分兼容）
- 考虑做 Edge 插件（共用同一套代码）
